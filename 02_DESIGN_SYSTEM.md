# Purpose

This document defines every visual rule of KHAS Lifestyle.

Nothing should be designed outside this system.

Every new UI component must follow these standards.

---

# Design Philosophy

KHAS Lifestyle should feel:

- Premium
- Modern
- Clean
- Calm
- Minimal
- Trustworthy
- Elegant

The interface should never feel crowded.

The design should guide attention naturally.

Whitespace is considered a design element.

---

# Design Principles

Always prefer:

Less UI

More Content

Less Decoration

More Readability

Less Noise

More Focus

Every pixel should have a purpose.

---

# Spacing System

Spacing follows an 8px grid.

Approved spacing scale:

4

8

12

16

24

32

40

48

64

80

96

Avoid arbitrary spacing values.

---

# Border Radius

Use consistent radius values.

Small

Medium

Large

Extra Large

Never mix random radius values.

---

# Shadows

Shadows must remain subtle.

No heavy shadows.

Prefer depth through spacing instead of blur.

---

# Layout Philosophy

Content width should remain comfortable.

Pages should breathe.

Avoid edge-to-edge content unless intentionally required.

---

# Grid

Desktop

Tablet

Mobile

Every section should naturally adapt.

No fixed desktop layouts.

---

# Responsive Philosophy

Mobile First

Tablet Second

Desktop Third

Responsive behavior should never be added later.

It is part of the initial implementation.

---

# Components

Every component must have:

Single responsibility

Reusable structure

Predictable behavior

Consistent spacing

Consistent typography

Consistent interaction

---

# Buttons

Buttons should feel:

Simple

Premium

Readable

Never oversized.

Never overly colorful.

Primary actions should be obvious.

Secondary actions should remain visually lighter.

---

# Forms

Forms should prioritize:

Clarity

Accessibility

Speed

Visible labels

Clear validation

Predictable focus states

---

# Cards

Cards are content containers.

Cards should not become decorative objects.

Use:

Soft borders

Minimal shadows

Comfortable spacing

Readable hierarchy

---

# Navigation

Navigation should always remain simple.

The user should never wonder:

"Where am I?"

"Where do I click?"

---

# Icons

Icons should be:

Simple

Consistent

Readable

Avoid mixing icon styles.

---

# Images

Images should support content.

Never replace content.

Always optimize.

Always compress.

Always provide alt text.

---

# Animations

Animations should be subtle.

Animation should communicate state.

Animation should never exist only for decoration.

Avoid excessive movement.

---

# Accessibility

Always maintain:

Readable contrast

Large touch targets

Keyboard accessibility

Visible focus states

Semantic HTML

Accessibility is mandatory.

---

# Visual Hierarchy

The user should naturally notice:

1. Main Heading

2. Primary Action

3. Featured Content

4. Supporting Content

Everything else follows.

---

# Empty States

Every empty state should explain:

What happened

Why

What the user can do next

---

# Error States

Errors should never blame the user.

Explain clearly.

Offer recovery.

Remain calm.

---

# Loading States

Always prefer skeleton loading over blank pages.

Loading should reduce uncertainty.

---

# Content Style

The interface should never compete with the content.

Content is the hero.

Design supports content.

---

# Future Updates

This document is expected to evolve.

Whenever typography, colors, or branding become finalized:

Update this document first.

Then update the implementation.

Documentation always comes before redesign.

---

# Final Principle

Good design is invisible.

Users should notice the content—not the interface.
